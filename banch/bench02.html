Test

Same test as bench01: 1,000,000 iterations of a discarded binary addition sum + 1, isolating per-iteration expression-eval cost (operand lookup + operator dispatch + result allocation).

// Sard
sum = 0
start = clock()
loop(1000000) {
    sum + 1
}
elapsed = clock() - start
print("add loop 1M: ", elapsed, "ms")
# Python
import time
sum = 0
start = time.time()
for _ in range(1000000):
    sum + 1
elapsed = (time.time() - start) * 1000
print("add loop 1M: ", int(elapsed), "ms")
-- Lua (bare expr not allowed; assign to throwaway local)
local sum = 0
local start = os.clock()
for i = 1, 1000000 do
    local _ = sum + 1
end
local elapsed = (os.clock() - start) * 1000
print("add loop 1M: ", math.floor(elapsed), "ms")
Note on Lua: Lua rejects bare expression statements (sum + 1), so the result is bound to a throwaway local _. This still measures the binary-op cost and is functionally equivalent to the Sard/Python versions.

Broader Sard context (single pass)

For orientation, here is where add_loop sits among the rest of the Sard micro-benchmarks on the same build. These are single-pass numbers (not medians) so treat them as ±5%.

Observations from this breakdown:

  • Empty loop = 125 ms — the floor. Loop infra + scope create/destroy costs ~125 ns per iteration. This is the irreducible cost of the current loop design.
  • ident read = 2453 ms — ~2.3 µs per sum lookup. This is the scope-chain walk + TStringList.IndexOf + AddRef, four scope hops for sum.
  • pure assign = 2594 ms — ~2.6 µs per sum = 1. Only ~140 ns more than a read, thanks to the SetMemberAt fast path added in the optimizations.
  • add_loop = 4110 ms — ~4.1 µs per sum + 1. The binary op costs ~1.7 µs over the bare ident read: one EvalLiteral + one NewValue for the result + one ApplyOperator dispatch + the result Release. The result-box allocation is the bulk of it.
  • bench_loops (sum += 1) = 6781 ms — ~6.8 µs, almost exactly ident(2453) + add(4110 − 2453) + assign overhead. The compound-assign path re-evaluates sum twice (once as lvalue, once as the binary left operand), which is visible here.

Review & Analysis (unchanged direction, confirmed numbers)

The retest confirms the bench01 finding with tighter statistics: Sard is ~3 orders of magnitude behind Lua and ~2 orders behind Python on a tight numeric loop. The relative gap is not a compiler-quality issue (Sard is built with FPC at -O3); it is the tree-walking + per-value heap-object model.

Per-iteration cost breakdown for sum + 1 in Sard (~4.1 µs):

  1. Tree-walking dispatch — EvalNode → EvalBinary → EvalIdentifier + EvalLiteral → ApplyOperator, recursive calls and a case on Node.Kind at every level. Lua/Python decode a single bytecode opcode instead.
  2. Per-result heap allocation — a fresh TSardValue is allocated for the sum + 1 result on every iteration and immediately freed (refcount → 0). Lua reuses a register slot; Python uses the value stack. Neither allocates a boxed object per arithmetic op.
  3. Scope-chain walk + TStringList.IndexOf — reading sum walks iteration-scope → loop-scope → outer-scope → root and does a linear case-insensitive string compare at each hop. Lua resolves sum as a pre-resolved upvalue slot; Python does LOAD_NAME against one dict.
  4. Refcount churn — every produced value gets AddRef/Release pairs along its eval path, even with the recent AddRef-instead-of-Clone optimization. Lua has no per-op refcounting; Python amortizes it under the GIL.

Where the realistic headroom is (confirmed)

  • A bytecode / VM layer — the single biggest win. Compile the AST to a flat opcode stream once, then interpret opcodes. Eliminates recursive tree-walking, enables pre-resolved variable slots, and turns ADD into one branch in a tight loop. This is what makes Lua ~2000x faster, not Lua's C vs. Sard's Pascal.
  • Avoid allocation for discarded results — sum + 1 as a statement has its result thrown away. An nkExprStmt wrapper that evaluates for side-effects only would skip boxing the result and cut roughly one alloc + one free per iteration here (estimated ~30-40% off add_loop alone).
  • Inline cache for EvalIdentifier — remember (scope-depth, member-index) of sum after the first lookup; subsequent lookups validate the cache and skip the chain walk. Standard monomorphic-IC technique; works well for loop bodies.
  • Unbox small integers — NaN-boxing or tagged-pointer scheme (like LuaJIT/JavaScriptCore) removes the per-value heap object entirely for numbers, killing the alloc+free churn on arithmetic.
Bottom line (retest confirms): Sard is a clean tree-walking interpreter with per-value heap objects and refcounting. That design is ~3 orders of magnitude slower than a bytecode VM for tight numeric loops. The recent optimizations were the right local wins (~2x on most benches, ~6x on empty loop) but closing the gap to Lua/Python requires moving off tree-walking to a compiled bytecode/VM layer plus value unboxing — a design change, not a micro-optimization. Numbers in this retest are within noise of bench01; no regression from the Delphi-compat commit.
