What changed

Since bench02, three commits landed:

  • f709caa — Fix Hashmap range error
  • 6ccc0ac — Cleaning
  • 40be858 — Rename vkLazy to vkRawArg

The first one is the performance story: member lookup moved from a TStringList (linear scan) to a hashmap. This turns every variable/member lookup from O(n) into ~O(1). The effect is dramatic on the benchmarks that repeatedly read variables.

Test

Same 1,000,000-iteration discarded binary-addition test as in bench01/bench02:

// Sard
sum = 0
start = clock()
loop(1000000) {
    sum + 1
}
elapsed = clock() - start
print("add loop 1M: ", elapsed, "ms")
# Python
import time
sum = 0
start = time.time()
for _ in range(1000000):
    sum + 1
elapsed = (time.time() - start) * 1000
print("add loop 1M: ", int(elapsed), "ms")
-- Lua
local sum = 0
local start = os.clock()
for i = 1, 1000000 do
    local _ = sum + 1
end
local elapsed = (os.clock() - start) * 1000
print("add loop 1M: ", math.floor(elapsed), "ms")

Key observations:

  • Lookup-bound benchmarks win the most. bench_ident_loop (pure reads) went 26x faster because it does almost nothing except lookup sum — exactly the operation the hashmap accelerates.
  • Assignment wins 13.9x. bench_pure_assign writes sum = 1; the write path also needs to find the existing slot (via hash) and replace it.
  • Compound assignment wins 4.0x. bench_loops is sum += 1 (parsed as sum = sum + 1). Two lookups per iteration (lvalue + rhs) benefit from O(1) hashing, plus the arithmetic/object overhead.
  • Arithmetic-heavy bench wins 3.7x. bench_arith still spends most time in binary-op result creation; hashmap removes the lookup tax, leaving the per-value allocation as the new dominant cost.
  • Empty loop wins 2.0x. Even with no variable access, scope creation/teardown is faster, likely because member containers are now hashmaps and are cheaper to construct/destroy than the previous TStringList setup.
  • Scope size no longer matters. The dedicated bench_scope_size test shows 1515 ms baseline and 1500 ms with 20 extra root members — effectively identical. This is the clearest proof that lookup is now O(1) rather than O(n).

Review & Analysis

The hashmap change is the single biggest interpreter performance improvement so far. It does not change Sard's tree-walking, per-value allocation, or refcounting models — it removes the lookup tax that those models were paying on every variable access. The result is a 2×–26× speedup depending on how lookup-bound the benchmark is.

Remaining dominant costs in Sard (after hashmap)

  1. Per-result heap allocation. Every sum + 1 still allocates a TSardValue object and frees it. This is now the largest remaining cost in add_loop.
  2. Tree-walking dispatch. Recursive AST traversal (EvalNode → EvalBinary → ...) still runs on every iteration. Bytecode VMs avoid this.
  3. Refcount churn. AddRef/Release on every evaluated subexpression still adds per-iteration overhead.

Where the next wins are

  • Avoid allocating discarded expression results. sum + 1 as a statement throws the result away. An nkExprStmt node that evaluates only for side effects would skip boxing the result and could cut a meaningful chunk off add_loop.
  • Bytecode / VM layer. Still the largest architectural win. Compile the AST once, then run a tight opcode loop. This eliminates recursive tree-walking and enables pre-resolved variable slots (slot lookups are even cheaper than hash lookups).
  • Value unboxing / NaN-tagging. Remove the wrapper object for integers/floats and store them inline or in tagged pointers, eliminating the alloc+free per arithmetic result.
  • Literal caching. The constant 1 in sum + 1 is re-evaluated (and a new value allocated) on every iteration. Caching small integer literals would save one alloc+free per loop turn.
Bottom line: The hashmap is a huge, correct local optimization that moved Sard from “3 orders of magnitude behind Lua” to “2.6 orders of magnitude behind Lua” on this loop. The gap to bytecode VMs is still real, but it is now dominated by allocation/dispatch models rather than lookup algorithms. The next biggest wins are bytecode compilation and value unboxing, not more lookup tricks.
