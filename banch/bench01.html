Benchmark 01 — add_loop

Date: 2026-06-18   Host: Windows x64   Sard: commit 6e606fd (FPC 3.3.1, -O3)

Test

Measure the overhead of a binary addition whose result is discarded, inside a 1,000,000-iteration loop. This isolates the cost of evaluating one binary expression node (operand lookup + operator dispatch + result allocation) per iteration.

// Sard
sum = 0
start = clock()
loop(1000000) {
    sum + 1
}
elapsed = clock() - start
print("add loop 1M: ", elapsed, "ms")
# Python
import time
sum = 0
start = time.time()
for _ in range(1000000):
    sum + 1
elapsed = (time.time() - start) * 1000
print("add loop 1M: ", int(elapsed), "ms")
-- Lua
local sum = 0
local start = os.clock()
for i = 1, 1000000 do
    local _ = sum + 1
end
local elapsed = (os.clock() - start) * 1000
print("add loop 1M: ", math.floor(elapsed), "ms")
Note on Lua: Lua does not allow a bare expression statement (sum + 1), so the result is assigned to a throwaway local _. This still measures the binary-op cost and is functionally equivalent to the Sard/Python versions.

Review & Analysis

The ~1350x gap between Lua and Sard is not a compiler-quality issue (Sard is already built with FPC at -O3); it is a fundamental architecture difference. The dominant costs in Sard's sum + 1 per iteration are:

  1. Tree-walking dispatch. Each iteration re-dispatches on Node.Kind through EvalNode → EvalBinary → EvalIdentifier + EvalLiteral → ApplyOperator. Lua and Python compile to bytecode once and the interpreter loop just decodes opcode ADD — no AST traversal, no recursive calls.
  2. Per-result heap allocation. Sard allocates a fresh TSardValue object for the sum + 1 result on every iteration, then immediately frees it (refcount drops to 0). That is one heap alloc + one heap free per loop turn. Lua uses a register-based VM and reuses stack slots; Python uses a value stack. Neither allocates a boxed object per arithmetic op.
  3. Scope-chain walk + member lookup. Reading sum walks the iteration-scope → loop-scope → outer-scope → root chain and does a TStringList.IndexOf at each hop. Lua resolves sum as an upvalue via a precomputed slot; Python resolves it via LOAD_NAME against a dict, but the dict is one level and the name is cached.
  4. Refcount churn. Even with the recent AddRef-instead-of-Clone optimization, every produced value still gets AddRef/Release pairs on each eval path. Lua has no per-op refcounting; Python relies on the GIL and amortized refcounting.

The recent Sard optimizations (lazy containers, AddRef for immutables, FindVariable returning a member index, SetMemberAt fast path) brought add_loop from ~5640 ms down to ~4040 ms — a ~1.4x improvement within the tree-walking model. Further wins inside the current model are diminishing: the remaining cost is the model itself.

Where the realistic headroom is

  • A bytecode / VM layer (the single biggest win). Compile the AST to a flat opcode stream once, then interpret opcodes. This eliminates recursive tree-walking, enables pre-resolved variable slots, and turns ADD into one branch in a tight loop. This is what makes Lua ~1350x faster, not Lua's C vs. Sard's Pascal.
  • Avoid allocation for discarded results. sum + 1 as a statement has its result thrown away. A compiler/VM pass can flag expression-statement results as "unused" and skip boxing them. In the current AST interpreter, EvalStatements always Releases the returned value — so we always pay for the box. A nkExprStmt wrapper that evaluates for side-effects only would cut roughly one alloc+free per iteration here.
  • Inline cache for EvalIdentifier. Remember the (scope-depth, member-index) of sum after the first lookup; subsequent lookups validate the cache and skip the chain walk. This is the standard monomorphic-IC technique and works well for loop bodies.
  • Unbox small integers in TSardValue. If Kind = vkInteger is stored inline (it already is — IntValue: Int64), the cost is the wrapper object, not the int. A NaN-boxing or tagged-pointer scheme (like LuaJIT/JavaScriptCore) would remove the per-value heap object entirely for numbers.
Bottom line: Sard is currently a clean tree-walking interpreter with per-value heap objects and refcounting. That design is ~3-4 orders of magnitude slower than a bytecode VM for tight numeric loops. The recent optimizations were worthwhile (~2x on most benches, ~6x on empty loop) and were the right local wins, but closing the gap to Lua/Python requires moving off tree-walking to a compiled bytecode/VM layer plus value unboxing — a design change, not a micro-optimization.
